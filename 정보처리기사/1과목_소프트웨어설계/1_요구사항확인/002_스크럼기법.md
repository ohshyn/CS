# 1 요구사항 확인

## 002 스크럼(Scrum) 기법

### [1] 스크럼의 개요

팀이 중심이 되어 개발 효율성을 높이는 애자일 모형

```
# 팀 중심의 개발 기법
	Self-Organizing: 팀원 스스로가 스크럼 팀을 구성
	Cross-Functional: 개발 작업에 관한 모든 것을 팀 스스로 해결

# 한 팀의 구성
	제품 책임자(PO; Product Owner)
	스크럼 마스터(SM; Scrum Master)
	개발팀(DT; Development Team)

# 제품 책임자(PO; Product Owner)
	주로 제품 이해도가 높은 개발 의뢰자 / 사용자
	이해관계자들의 의견을 종합해 제품의 요구사항을 작성
	요구사항 백로그(Backlog)*를 작성, 백로그의 우선순위를 지정
	팀원들은 스토리만 추가 가능
	제품 테스트를 수행하며 요구사항 우선순위를 주기적으로 갱신

# 스크럼 마스터(SM; Scrum Master)
	스크럼 팀이 스크럼을 잘 수행하도록 조언하는 가이드, 통제하는 것이 목표가 아님
	일일 스크럼 회의를 주관하여 진행 사항 점검, 개발 과정의 발생 장애 요소를 공론화하여 처리

# 개발팀(DT; Delelopment Team)
	제품 책임자, 스크럼 마스터를 제외한 나머지 팀원
	보통 최대 7~8명
```

```
* 백로그(Backlog)
	요구사항들 + 각각의 우선순위
	제품 개발에 필요한 요구사항을 모아 우선순위를 부여한 목록

* 스토리(Story)
	백로그에 담길 요구사항
```

### [2] 스크럼 개발 프로세스

[제품 백로그] > [스프린트 계획 회의] > [반복: [스프린트] > [매일: [일일 스크럼 회의]] > [스프린트 검토 회의] > [스프린트 회고]]

```
# 제품 백로그(Product Backlog)
	모든 요구사항 + 우선순위
	새롭게 도출되는 요구사항으로 지속 업데이트
	스토리를 기반으로 전체 일정 계획(Release Plan, 릴리즈 계획)을 수립

# 스프린트 계획 회의(Sprint Planning Meeting)
	백로그 중 이번 스프린트에서 진행할 작업을 대상으로 단기 일정 수립
	스토리를 태스크(Task, 작업단위)로 분할한 후 스프린트 백로그(Sprint Backlog, 개발자 별 태스크 목록)을 작성

# 스프린트(Sprint)
	보통 2~4주의 실제 개발 작업
	스프린트 백로그 내 태스크의 작업 시간을 추정하여 담당자에게 할당
	태스크는 할 일(To Do), 진행 중(In Progress), 완료(Done)의 상태를 가짐

# 일일 스크럼 회의(Daily Scrum Meeting)
	매일 약속된 시간에 15분의 짧은 시간 동안 진행 상황을 점검
	남은 작업 시간을 소멸 차트(Burn-down Chart)*에 표시
	스크럼 마스터는 장애요소 해결을 도움

# 스프린트 검토 회의(Sprint Review)
	스프린트 결과물이 요구사항에 부합하는지 테스팅
	스프린트의 한 주동 한 시간 내에 진행
	제품 책임자는 피드백을 정리하고 제품 백로그를 업데이트

# 스프린트 회고(Sprint Retrospective)
	스프린트 주기 진행동안 개선 사항을 확인, 기록
	일정 주기로 수행
```

```
* 소멸 차트(Burn-down Chart)
	해당 스프린트에서 수행할 작업 진행 상황을 확인 가능하도록
	시간에 따라 남은 작업 시간을 그래프로 표현한 것
```
